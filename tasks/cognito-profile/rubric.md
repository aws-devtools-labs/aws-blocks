auth_correctness — Does the implementation run a real passwordless email-OTP flow on the Cognito auth block (sign-up/sign-in sends a one-time code, confirming it establishes a session), surface the signed-in user's identity on a profile view, reject wrong codes gracefully, restore the session on reload, and support a clean sign-out — without ever using a password? Is the wrong-code path correctly retriable — a rejected code leaves the verification session valid so the correct code can be entered next on the same view, and the `auth-error` hook is cleared once sign-in succeeds? Does an empty/whitespace-only email get validated away (no auth begun, no advance to code entry)? Across repeated sign-in/out cycles does the profile always show the current identity with no stale/cached user, and does signing out a cookie-RESTORED session (after a reload) fully clear it (calling the block's signOut, not merely flipping a client flag)? Score DOWN if errors aren't cleared on a successful retry, if blank emails begin auth, or if a prior identity leaks across cycles.
